# Приглашение инфлюенсера / реферала в клиентское приложение

Пошаговая инструкция для агентства. Предполагается, что в Supabase уже применены миграции реферала и `client_party.show_referral_in_app`.

---

## A. Что подготовить в CRM (справочник)

1. **Карточка контакта** (person или company) с **email**, который инфлюенсер будет вводить при входе в приложение.
2. Роль **Client** (обязательно для `client_profiles` и входа по email/паролю).
3. Галочка **Show referral section in client app** (роль Client → блок под ролями).
4. Роль **Referral** и **ставки по категориям** — иначе в приложении не появятся суммы и строки начислений (будет предупреждение и пустые блоки).
5. На **заявках** при необходимости укажите этого контакта как реферала и логику подтверждения — как у вас настроено в CRM.

Пока нет строки в `client_profiles`, вход в приложение **невозможен** — сначала нужна регистрация по приглашению (или ручное создание профиля в БД, не описываем здесь).

---

## B. Секреты на сервере (Vercel / `.env.local`)

Должны совпадать у **Next.js API** и у **скрипта приглашения** на вашей машине:

- `CLIENT_JWT_INVITATION_SECRET` — длинная случайная строка (не оставляйте dev-fallback в проде).

Также для приложения:

- `EXPO_PUBLIC_API_URL` — URL продакшен-API (например `https://your-domain.com`).

---

## C. Режим «только реферал» (сырая сборка приложения)

Чтобы в приложении **не показывались** Home, Trips, Concierge, Documents и лишние вкладки:

В **Expo / EAS** задайте при сборке:

```bash
EXPO_PUBLIC_CLIENT_APP_REFERRAL_ONLY=1
```

(или `true`)

Тогда после входа открывается **только экран Referral** и кнопка **Sign out**. Логин/регистрация те же.

Когда полный клиентский продукт будет готов — уберите эту переменную или поставьте `0`.

---

## D. Выпустить приглашение (JWT на 24 часа)

1. Скопируйте **UUID карточки** из CRM: `party.id` (тот же id в URL детальной страницы справочника).

2. (Опционально) UUID агента из **Supabase → Authentication → Users** (`user_id`), который есть в `public.profiles` — попадёт в `invited_by_agent_id`. Если не указать — в БД будет `NULL`, это нормально.

3. В корне репозитория **travel-cms** (где лежит `.env.local` с тем же `CLIENT_JWT_INVITATION_SECRET`, что и на сервере):

```bash
npx tsx scripts/generate-client-app-invite.ts <PARTY_UUID> [AGENT_AUTH_USER_UUID]
```

Скрипт выведет:

- сам **JWT**;
- **deep link** вида  
  `mytravelconcierge://register?invitationToken=...`

4. Отправьте инфлюенсеру:

   - ссылку **или** сам JWT (если откроете регистрацию вручную);
   - инструкцию: установить сборку приложения → открыть ссылку на телефоне **или** пройти экран регистрации с токеном;
   - задать **пароль не короче 8 символов**;
   - дальше вход: **тот же email, что в карточке CRM**, и этот пароль.

Срок действия ссылки: **24 часа**. Потом сгенерируйте новый токен тем же скриптом.

---

## E. Deep link

Схема приложения: **`mytravelconcierge`** (`Client/app.config.ts`).

Пример:

`mytravelconcierge://register?invitationToken=<JWT>`

На симуляторе iOS для проверки:

```bash
xcrun simctl openurl booted "mytravelconcierge://register?invitationToken=PASTE_JWT_HERE"
```

Если ссылка не открывается, используйте **ручной** путь: добавьте в приложении переход на Register и передачу токена (при необходимости — отдельная доработка).

---

## F. Контрольный чеклист

| Шаг | Действие |
|-----|----------|
| 1 | Email в карточке = будущий логин |
| 2 | Роль Client + галочка referral в app |
| 3 | Роль Referral + ставки (для цифр) |
| 4 | `CLIENT_JWT_INVITATION_SECRET` на сервере и локально для скрипта |
| 5 | Сгенерировать invite, отправить ссылку |
| 6 | Инфлюенсер: регистрация → пароль → вход |
| 7 | `EXPO_PUBLIC_API_URL` указывает на нужный backend |

---

## G. Повторный вход

После регистрации используется только **Login** (email + пароль). Новое приглашение нужно, если профиль ещё не создан или вы хотите сменить связку (обычно не требуется).
